# Rules — Prioritisation

Confirmed patterns to apply by default. Demote to `HYPOTHESES.md` if later contradicted.

1. Use RICE to rank already-validated candidate initiatives against each other, not to decide whether an initiative is worth doing at all. RICE presumes value risk has already been addressed through discovery.

2. Treat a low Confidence score in a RICE estimate as an explicit instruction to run an assumption test before trusting the Reach or Impact figures, rather than as a minor caveat. Confidence exists precisely to catch inflated estimates made without evidence.

3. Use MoSCoW when the constraint is a fixed delivery date or fixed resource window and the question is what minimum scope satisfies it, not when the question is which of several options is most valuable. Do not use MoSCoW as a substitute for value ranking.

4. When running a MoSCoW exercise, cap the "Must have" category deliberately (a common convention is roughly 60% of total effort) and push back when every stakeholder claims their item is a Must. If most items land in "Must," the categorisation has failed to create real tradeoffs.

5. Before prioritising a portfolio of features, classify at least the contested ones using Kano categories (basic, performance, excitement) rather than assuming impact uniformly. A feature wrongly assumed to be a delighter that is actually a basic expectation will be systematically overweighted in RICE-style scoring.

6. Deliberately balance a roadmap across Kano categories rather than optimising a single one: basic features alone will not differentiate the product, and excitement features alone risk leaving baseline expectations (whose absence causes disproportionate dissatisfaction) unmet.

7. Re-run Kano classification periodically for features that anchor competitive differentiation, since excitement features migrate toward performance and then basic as a market matures and competitors converge on the same capability.
