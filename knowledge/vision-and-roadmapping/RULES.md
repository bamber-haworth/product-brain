# Rules — Vision and Roadmapping

Confirmed patterns to apply by default. Demote to `HYPOTHESES.md` if later contradicted.

1. Fill in a Product Vision Board's target group and needs fields before the product description field. If the product description is drafted first, the needs and target group fields tend to be retrofitted to justify an already-decided feature set rather than derived independently.

2. Treat the Product Vision Board as a living document to be revisited as evidence accumulates, not a one-time artifact authored and archived at project kickoff.

3. Write roadmap entries as goals with a timeframe and, where possible, a success metric, not as named features with ship dates. A roadmap item with no stated metric for success is a weaker version of the format and should be tightened before being published.

4. Before adding an item to the roadmap, check it traces back to a stated need or business goal on the Product Vision Board. An item with no visible link back is a signal of scope drift.

5. When a goal-oriented roadmap must be shared with stakeholders who expect fixed feature commitments and dates (e.g., sales, contractual, or marketing-launch contexts), produce a separate, more concrete release plan derived from the roadmap once solutions are chosen, rather than presenting the goal-oriented roadmap itself as a delivery commitment.

6. Use coarse, named horizons (e.g., now/next/later) rather than fixed calendar quarters for roadmap items far enough out that specific dates would overstate certainty.
